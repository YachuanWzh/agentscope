# AgentScope 的 ReAct 模式（推理-行动循环）大白话解读

## 一、什么是 ReAct？

**ReAct = Reasoning（推理）+ Acting（行动）**。这是一个让 AI Agent 能"边想边干"的核心工作模式。

打个比方：你让 Agent 帮你查天气、订酒店、写报告。Agent 不是一次性把活干完的，而是：

1. **推理（Reasoning）**：调用大模型想一想——"用户要查天气，我应该调天气工具，但我需要知道城市名，他好像没说……"
2. **行动（Acting）**：执行工具调用——调用天气 API，拿到结果
3. **再推理**：把工具返回的结果喂给大模型——"结果显示北京今天 25 度，我把这个告诉用户吧"
4. **再行动**：如果有新工具要调就继续，没有就输出最终回复

这个"思考 → 干活 → 看结果 → 再思考 → 再干活 → ……"的循环，就是 ReAct 模式的核心。

## 二、AgentScope 的 ReAct 怎么设计的？

在 AgentScope 里，整个 Agent 是一个叫 `Agent` 的类（在 `src/agentscope/agent/_agent.py` 里）。它的 ReAct 循环不是抽象概念，而是实实在在一个 `while` 循环：

```python
while self.state.cur_iter < self.react_config.max_iters:
    # 决定下一步做什么
    action, data = self._check_next_action()

    if action == "exit":
        yield data  # 等待外部交互，先退出去
        return

    if action == "reasoning":
        # 调用大模型：想一想
        async for evt in self._reasoning():
            if isinstance(evt, Msg):
                # 大模型没产生工具调用，直接输出回复，结束
                yield evt
                return
            yield evt

    # 执行工具调用：干活
    for batch in await self._batch_tool_calls():
        # 逐个批次执行……
        ...

    self.state.cur_iter += 1  # 记一次循环，防止死循环
```

### 2.1 三层架构：Reply → Reasoning → Acting

整个 Agent 的工作流程分三个层级，一层包一层：

```
┌─────────────────────────────────────────────────┐
│  _reply（回复入口）                                │
│  ┌───────────────────────────────────────────┐   │
│  │  _reasoning（推理 = 调用大模型）             │   │
│  │  - 准备消息和工具列表                       │   │
│  │  - 调用 LLM，拿到文本回复 + 工具调用请求     │   │
│  │  - 如果没有工具调用 → 直接输出最终回复        │   │
│  └───────────────────────────────────────────┘   │
│                      ↓                           │
│  ┌───────────────────────────────────────────┐   │
│  │  Acting（行动 = 执行工具）                  │   │
│  │  - 批量组织工具调用（顺序/并发）             │   │
│  │  - 权限检查（要不要问用户？）               │   │
│  │  - 执行工具并将结果写回上下文               │   │
│  └───────────────────────────────────────────┘   │
│                      ↓                           │
│           回到 _reasoning，再来一轮               │
└─────────────────────────────────────────────────┘
```

### 2.2 状态机：三个动作，精确决策

`_check_next_action()` 是一个纯逻辑状态机，不涉及大模型调用。它根据当前上下文中的工具调用状态来决定下一步：

|                          | 有待处理的外部交互         | 没有待处理的外部交互   |
| ------------------------ | -------------------------- | ---------------------- |
| **有可执行的工具调用**   | **acting**（先执行能执行的） | **acting**（干活）      |
| **没有可执行的工具调用** | **exit**（等外部操作完成）   | **reasoning**（继续想） |

三种动作的含义：

- **exit**：Agent 有工具调用在等用户确认或外部执行，没别的可做，先退出循环，把控制权交还给外界。
- **reasoning**：手上没活干、也没在等人，那就继续问大模型"下一步该干嘛"。
- **acting**：有已经审批通过的工具调用要执行，直接干活。

这个状态机非常关键——它让 ReAct 循环不需要大模型来管"该进入哪个阶段"，而是通过代码逻辑精确控制，避免了用 token 去判断控制流的浪费。

### 2.3 工具调用状态生命周期

每个工具调用都有完整的生命周期状态，用 `ToolCallState` 枚举来追踪：

```
大模型产生工具调用
       ↓
   PENDING（待处理）
       ↓
   权限检查
   ├── ALLOWED → 执行 → FINISHED
   ├── ASKING → 等用户确认 → ALLOWED/DENIED
   └── SUBMITTED → 等外部执行 → FINISHED
```

- **PENDING**：大模型刚生成的工具调用，还没决定怎么处理
- **ALLOWED**：权限通过了，可以执行
- **ASKING**：需要弹窗问用户"我能执行这个操作吗？"
- **SUBMITTED**：已提交给外部系统执行（比如在用户本地运行命令），等结果中
- **FINISHED**：执行完成，结果已写入上下文

这个状态追踪的好处是：即使对话被中断、上下文被压缩、用户半天才回，Agent 也能准确知道"我刚才哪个工具调用做到哪一步了"。

### 2.4 工具调用批处理：聪明地并发

Agent 在拿到大模型返回的一批工具调用后，不是一股脑全并发或全顺序执行。它通过 `_batch_tool_calls()` 把工具调用分成多个批次：

- **并发批次**：工具标记了 `is_concurrency_safe=True`（不会互相影响），或者工具没注册（马上会报错，不影响其他），归到一个并发批次里同时执行
- **顺序批次**：有副作用、不安全的工具（如写文件、删除数据），按顺序一个一个来

这种设计兼顾了**效率**（能并发的绝不排队）和**安全**（有副作用的必须按序执行）。

## 三、ReAct 为什么这么设计？

### 核心问题 1：大模型会"无限死循环"

大模型有时候会不停地调工具、看结果、再调工具、再看结果……陷入一个无底洞。比如让它"优化代码性能"，它可能永远觉得"还能再改改"。

**解决方案**：`ReActConfig.max_iters`（默认 20 轮）。这是一个硬上限——Agent 的推理-行动循环最多跑 20 轮，超过就强制停止并告诉用户"我执行了最大轮次但没完成"。这个值可以通过配置调整。

### 核心问题 2：上下文窗口会"撑爆"

每次推理都把对话历史 + 工具调用 + 工具结果全发给大模型，上下文窗口很快就满了，Agent 就会"失忆"。

**解决方案**：上下文压缩机制。在每次 reasoning 之前，Agent 会检查当前 token 数是否超过阈值（默认是模型上下文长度的 80%），如果超过了，就触发压缩：

1. 把旧的对话历史用另一个模型调用压缩成结构化摘要（包含任务概览、当前进度、重要发现、下一步计划）
2. 旧对话被替换成摘要，token 大大减少
3. 摘要里专门保留了任务进度信息，Agent 不会"忘记自己在干嘛"

这个设计和 Plan 模式（Task 工具）形成了互补：Task 工具是 Agent 主动记录的结构化计划，上下文摘要是被动压缩时的保底记忆。

### 核心问题 3：有些工具操作需要人审批

Agent 可能要删文件、发邮件、执行命令——这些操作如果全自动执行，风险太大。

**解决方案**：权限引擎（`PermissionEngine`）内置在工具执行流程中。每个工具调用在执行前都要过权限检查：

- `ALLOW`：直接放行
- `ASK`：弹窗问用户"同意吗？"——Agent 暂停循环，等人回复
- `PASSTHROUGH`：把决定权交给上层（比如前端 UI 自己判断）
- `DENY`：直接拒绝，告诉 Agent "不能干这个"

当权限结果是 `ASK` 时，ReAct 循环会**主动暂停**，退出 `_reply`，等用户确认后再从断点继续。这个"暂停-恢复"机制是 ReAct 循环支持人机协同的关键。

### 核心问题 4：Agent 不够灵活，外部想插一手怎么办？

不同场景下，用户可能想在 Agent 推理前加日志、在工具调用时做限流、在模型调用时换模型……如果所有逻辑都写死在 `_reply_impl` 里，代码会变成一锅粥。

**解决方案**：中间件系统（Middleware）。ReAct 循环在 5 个关键节点留了钩子：

| 钩子 | 位置 | 做什么用的 |
| ---- | ---- | ---------- |
| `on_reply` | 整个回复流程 | 最外层拦截，可以完全替换回复逻辑 |
| `on_reasoning` | 推理阶段（调大模型） | 在 LLM 调用前后做日志、缓存、内容审核 |
| `on_acting` | 行动阶段（调工具） | 限流、重试、工具调用日志、远程执行 |
| `on_model_call` | 原始模型 API 调用 | 模型切换、fallback、token 统计 |
| `on_system_prompt` | 系统提示词生成 | 动态修改 system prompt |

中间件用**洋葱模式**（onion pattern）：每个中间件包着下一个，执行顺序是 `mw1 前 → mw2 前 → 核心逻辑 → mw2 后 → mw1 后`。这跟 Web 框架（Express、Koa）的中间件思路一模一样。

代码里的实现方式非常优雅——通过 `next_handler` 回调实现链式调用：

```python
async def execute_chain(index=0, **kwargs):
    if index >= len(middlewares):
        # 链条走到头了，执行真正的逻辑
        async for item in self._reasoning_impl(**kwargs):
            yield item
    else:
        mw = middlewares[index]
        # 当前中间件可以决定什么时候调用 next_handler
        async for item in mw.on_reasoning(
            agent=self,
            input_kwargs=kwargs,
            next_handler=lambda **kw: execute_chain(index + 1, **kw),
        ):
            yield item
```

### 核心问题 5：怎么让整个系统支持流式输出？

用户不等 Agent 全部想完才看到结果——Agent 在推理和行动过程中就不断往外吐事件。

**解决方案**：一切皆事件流。`_reply_impl`、`_reasoning_impl`、`_execute_tool_call` 全都是 `AsyncGenerator`，不断地 `yield` 各种事件：

- `ModelCallStartEvent` / `TextBlockDeltaEvent` / `ModelCallEndEvent`（模型调用进度）
- `ToolCallStartEvent` / `ToolResultTextDeltaEvent` / `ToolResultEndEvent`（工具执行进度）
- `RequireUserConfirmEvent`（需要用户确认）
- `ReplyStartEvent` / `ReplyEndEvent`（回复开始/结束）

前端可以实时渲染——用户能看到 Agent 在想什么、在调什么工具、工具返回了什么，体验是"活的"而不是"卡住不动然后突然全出来"。

## 四、大白话总结

**ReAct 就是一个 "想一想 → 干一干 → 看结果 → 再想一想" 的死循环，但给它加了紧急刹车和很多安全措施。**

- **循环本身**：while 循环，每次问大模型"接下来干嘛"，大模型可能直接回答，也可能说要调工具；如果要调工具就去执行，把结果再喂给大模型继续问。

- **防死循环**：`max_iters` 硬上限，默认跑 20 轮就强制停止。

- **防失忆**：上下文太长就自动压缩成摘要，保留"我在干嘛、做了啥、下一步要干嘛"的关键信息。

- **防瞎搞**：工具调用前要过权限检查，危险操作弹窗问用户。人在中间打断也没关系，Agent 能"暂停-恢复"。

- **防慢**：工具调用能并发的并发跑，有副作用的排队跑。

- **防僵化**：5 个中间件钩子，外部可以随时插手修改 Agent 行为，不用改 Agent 核心代码。

- **防无聊**：全程流式输出事件，前端实时展示推理过程和工具执行进度。

**一句话：AgentScope 的 ReAct 不是一个简单的 "调 LLM → 执行工具" 循环，而是一个带有状态管理、权限控制、上下文压缩、中间件扩展、流式输出的完整 Agent 运行时。**
